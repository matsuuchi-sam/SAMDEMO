# トラブルシューティング — 問題と解決策まとめ

## 1. ESP32 関連

### 1.1 COMポートが見つからない
- **症状**: `pio run --target upload` で接続エラー
- **原因**: COMポート番号が変わった
- **解決**: デバイスマネージャーでCOMポート確認、`platformio.ini` の `upload_port` / `monitor_port` を更新

### 1.2 SPIFFS書き込みでPermissionError
- **症状**: `pio run --target uploadfs` で権限エラー
- **原因**: シリアルモニタなど他のプロセスがCOMポートを占有
- **解決**: モニタを閉じる、またはUSBケーブルを挿し直す

### 1.3 ダッシュボードが「--」のまま表示されない
- **症状**: ブラウザでページは表示されるがデータなし
- **原因**: Chart.js をCDNから読み込もうとしている。ESP32のWiFi APはインターネット接続がないためCDNは使えない
- **解決**: `chart.min.js` をダウンロードして `data/` フォルダに配置、HTMLを `<script src="/chart.min.js"></script>` に変更。SPIFFS再書き込み必須
- **教訓**: AP環境では外部CDN/Webフォントは一切使えない。すべてSPIFFSにローカル配信する

### 1.4 WebSocketが再接続を繰り返す
- **症状**: 接続→切断→再接続のループ
- **原因**: ESP32のメモリ不足、またはWebSocket接続数超過
- **解決**: AsyncWebServerのクリーンアップ確認。同時接続数を制限

## 2. GR-SAKURA (RX63N) 関連

### 2.1 rfp-cli で BootCode:80 エラー
- **症状**: 書き込み時に接続エラー
- **原因**: デバイスがブートモードに入っていない
- **解決**:
  1. スライドスイッチを **WRITE** に切り替え
  2. **USBケーブルを挿し直す**（SW1リセットだけでは不十分）
  3. 再度 `make flash` を実行

### 2.2 新規ビルドコードが動作しない (Issue #8)
- **症状**: 旧ビルド済みfirmware.motは正常動作するが、main.cを変更して再ビルドしたコードが動作しない
- **状態**: 未解決（根本原因不明）
- **回避策**: 旧プロジェクト (iot-demo-rx) のコードベースをそのまま使用。再ビルド時はフルアプリコードごとビルドする（最小コードだけだと動作しない）
- **テスト結果**:

| テスト | 結果 |
|--------|------|
| 旧firmware.mot（フルアプリ）書き込み | ✅ 動作 |
| 旧プロジェクト再ビルド（フルアプリ） | ✅ 動作（CRC一致） |
| main.cのみ最小LED点滅に置換 | ❌ 動作しない |
| gc-sections除去、旧リンカ使用 | ❌ 改善せず |

- **仮説**: コンパイラ最適化によるdelay除去、リンク時のセクション配置の問題、またはクロック起動のタイミング問題

### 2.3 JSONパーサーで "cmd" キーが見つからない
- **症状**: ブラウザから目標温度設定や緊急停止が効かない
- **原因**: `{"type":"cmd","cmd":"set_target","sp":30}` を解析する際、`find_value("cmd")` が `"type":"cmd"` の **値** の `"cmd"` に最初にマッチし、キーの `"cmd"` を見つけられない
- **解決**: `find_value()` で `"key"` を見つけた後、直後にコロン `:` があるかチェック。なければ値側のマッチとしてスキップし、次のマッチを探す

```c
/* 修正後: コロンの有無でキーと値を区別 */
while ((p = find_str(start, search)) != NULL) {
    const char *after = p + i;
    while (*after == ' ') after++;
    if (*after == ':') {
        after++;
        while (*after == ' ') after++;
        return after;
    }
    start = p + 1;  /* 値側のマッチ → 次を探す */
}
```

### 2.4 PID制御で目標温度超過後もPWMが下がらない
- **症状**: 温度31.5℃ > 目標28.0℃ なのに PWM=70/255
- **原因**: 積分項のワインドアップ。INTEGRAL_MAX=500000 が大きすぎて、加熱中に蓄積した積分値が残り続ける
- **解決**:
  1. 積分上限を縮小: 500000 → 50000
  2. 積分下限を0に設定（加熱のみシステムなので負の積分は不要）
  3. 目標温度超過時（error < 0）に積分値を即クリア

```c
#define INTEGRAL_MAX  50000L
#define INTEGRAL_MIN  0L

/* 目標超過時は積分クリア */
if (error < 0)
    pid->integral = 0;
```

### 2.5 SCI2の受信データ損失
- **症状**: コマンドが時々無視される
- **原因**: SCI2は1バイト受信バッファしかない。`sci0_puts()` でctrl JSON送信中（~5ms）に次のデータが到着すると、受信バッファがオーバーフローしてデータ損失
- **現状**: 軽微な問題（コマンド再送で回避可能）
- **改善案**: SCI2受信割り込みを有効化してリングバッファに蓄積、または `sci0_putc()` 内で送信待ち中に受信チェック

## 3. 通信関連

### 3.1 ESP32⇔GR-SAKURA UART接続
- **ピン接続**:
  - ESP32 GPIO17 (TX) → GR-SAKURA P52 (RXD2)
  - ESP32 GPIO16 (RX) ← GR-SAKURA P50 (TXD2)
  - GND共通接続必須
- **注意**: P20/P21（SCI0）では動作しなかった。P50/P52（SCI2）を使用すること
- **ボーレート**: 115200bps 8N1（両側で一致必須）

### 3.2 sci0_uart.c のファイル名と中身の不一致
- **注意**: ファイル名は `sci0_uart.c/h` だが、中身は **SCI2** レジスタ（P50/P52）を使用している。歴史的経緯によるもの。混乱を避けるため注意

## 4. 開発プロセスの教訓

### 4.1 ESP32 AP環境の制約
- CDN、Webフォント、外部APIは一切使えない
- すべてのリソースをSPIFFS（4MB以内）にローカル配信
- chart.min.js (208KB) のような大きなライブラリもローカル配置が必要

### 4.2 GR-SAKURA書き込みの注意点
- **必ずUSBケーブルを挿し直す**（SW1リセットだけでは不十分な場合が多い）
- WRITEモード⇔RUNモード切替時も挿し直す
- `-noverify-id` フラグは必須
- GR-SAKURAはCOMポートではなくUSB Direct接続（デバイスマネージャーに表示されない場合がある）

### 4.3 JSON解析の注意
- 同じ文字列がキーと値の両方に出現する場合がある（例: `"type":"cmd"` と `"cmd":"set_target"`）
- 単純な文字列検索ではなく、コロンの有無でキーを判別する必要がある

### 4.4 PID制御のチューニング
- 加熱のみ（冷却機能なし）のシステムでは、積分項の負方向は不要
- 目標温度超過時に積分項をクリアすることでオーバーシュート後の収束を早める
- D項（微分）を適切に設定することで、目標温度手前でPWMを下げる予測制御が可能
